# 编排算法问题分析与改进建议

## 一、当前系统概述

### 1.1 业务流程

```
用户输入 → 异常检测(CFM/Jigsaw) → QoS翻译 → 资源编排 → 分配结果
```

- **异常检测**: 根据输入数据（图像+点云 或 视频）计算异常分数
- **QoS翻译**: 根据异常分数生成QoS向量
- **资源编排**: 使用PPO强化学习算法分配节点和链路资源

### 1.2 核心组件

| 组件 | 文件 | 功能 |
|------|------|------|
| 感知服务 | `services/perception_service.py` | 异常检测 + QoS翻译 |
| 编排服务 | `intelligent_orchestration/orchestration_service.py` | 资源分配 |
| VNE环境 | `intelligent_orchestration/vne_env.py` | 强化学习环境 |
| PPO代理 | `intelligent_orchestration/ppo_agent.py` | 策略网络 |

### 1.3 当前QoS向量含义

| 索引 | 含义 | 说明 |
|------|------|------|
| 0 | 带宽(Bandwidth) | 网络带宽需求 0-1 |
| 1 | 延迟(Latency) | 延迟容忍度 0-1 |
| 2 | 优先级(Priority) | 服务优先级 0-1 |
| 3 | 丢包率(Loss Rate) | 丢包容忍度 0-1 |

## 二、当前问题分析

### 2.1 QoS参数未实际用于编排 ❌

**问题描述:**
- QoS向量被传入环境，但在奖励函数中的使用方式与业务逻辑不符
- 当前逻辑：QoS向量直接用于计算奖励，但权重固定，无法根据实际业务需求调整

**具体表现:**
```python
# vne_env.py 第314行
req_bw, req_lat, req_prio, req_loss = self.current_vnr.qos_vector
# req_prio (优先级) 完全未使用
# 带宽和延迟的计算方式与业务需求不匹配
```

**问题影响:**
- 用户无法通过调整QoS参数影响编排结果
- 参数变得无实际意义，增加系统复杂度

---

### 2.2 环境未感知动态资源变化 ❌

**问题描述:**
- 环境初始化时加载资源状态，但编排过程中不会更新
- 资源被占用/释放后，环境状态不会同步更新
- 每次编排请求可能看到过时的资源状态

**具体表现:**
```python
# vne_env.py - 只在初始化时加载
def __init__(self, node_count: int = 5, link_count: int = 8, use_db: bool = True):
    if use_db:
        db_nodes, db_links = load_resources_from_db()
        self.nodes = db_nodes  # 只加载一次
```

**问题影响:**
- 并发编排时可能出现资源冲突
- 同一资源可能被多次分配

---

### 2.3 链路资源未被分配 ❌

**问题描述:**
- 编排结果只包含节点分配，链路带宽始终为0
- 链路状态中的allocations始终为0

**具体表现:**
```python
# 编排返回结果
{
    "vnf_node": [2, 4],      # 节点分配
    "link_path": [],          # 链路为空！
    "node_allocation": {...}
}
```

**数据库链路状态:**
```
Link 0: free_bw=1.2, allocations=0  # 从未被占用
```

**问题影响:**
- 链路资源完全浪费
- 无法保证服务间的网络连通性
- 无法根据QoS的带宽需求分配链路

---

### 2.4 VNF数量固定为2 ❌

**问题描述:**
- VNF（虚拟网络功能）数量硬编码为2个
- 每个编排请求都会部署2个VNF，无法调整

**具体表现:**
```python
# vne_env.py 第125行
self.vnf_count = 2  # 硬编码

# action_space
action_dims = [self.node_count] * self.vnf_count  # 固定2个动作
```

**问题影响:**
- 无法支持单VNF部署场景
- 无法支持多VNF（如3个以上）部署
- 灵活性差

---

### 2.5 重复记录问题 ❌

**问题描述:**
- 一次编排可能产生两条历史记录
- 原因：step5的localStorage和后端数据库双重存储

**问题影响:**
- 历史记录显示混乱
- 已修复：删除step5的localStorage保存

## 三、系统架构问题

### 3.1 资源模型不合理

| 资源类型 | 当前状态 | 问题 |
|----------|----------|------|
| 节点CPU | 已分配 | 正常 |
| 节点Memory | 已分配 | 正常 |
| 链路带宽 | 未分配 | 链路资源完全未使用 |
| 链路延迟 | 已有数据 | 未用于编排决策 |

### 3.2 数据库设计问题

```sql
-- t_service_allocation 表
node_allocation: 节点分配信息 ✓
link_allocation: 链路分配信息 ✗ (未实现)
```

### 3.3 前端交互问题

- 编排请求的resource_request固定为 `{vcpu: 2, memory: 2}`
- 未使用户能够指定需要部署的VNF数量

## 四、改进建议

### 4.1 简化或重新设计QoS机制

**方案A：移除优先级参数**
- 保留带宽、延迟、丢包率三个有实际意义的参数
- 移除"优先级"参数（当前未使用）

**方案B：重新定义QoS含义**
| 参数 | 新含义 | 用途 |
|------|--------|------|
| 带宽 | 最低带宽需求 | 选择足够带宽的链路 |
| 延迟 | 最大延迟容忍 | 选择路径跳数少、延迟低的链路 |
| 丢包率 | 最大丢包容忍 | 选择高质量链路 |

**建议优先级: 高**

---

### 4.2 实现动态环境感知

**改进方案:**
```python
class GraphVNEEnv(gym.Env):
    def __init__(self, ...):
        # 保留初始化加载
        self._load_resources()

    def _load_resources(self):
        """每次编排前重新加载资源状态"""
        db_nodes, db_links = load_resources_from_db()
        self.nodes = db_nodes
        self.links = db_links

    def step(self, action):
        # 执行编排前先刷新资源状态
        self._load_resources()
        # ... 原有逻辑
```

**建议优先级: 高**

---

### 4.3 实现链路资源分配

**需要修改的文件:**

1. **vne_env.py**
   - 添加链路分配逻辑
   - 计算VNF间的链路需求
   - 更新链路带宽状态

2. **orchestration_service.py**
   - 保存链路分配记录到数据库

3. **数据库表**
   - t_service_allocation 添加 link_allocation 字段

**实现思路:**
```
1. 根据VNF部署位置确定通信需求
2. 计算节点间所需链路
3. 检查链路带宽是否足够
4. 预留链路资源
5. 释放时同时释放链路资源
```

**建议优先级: 高**

---

### 4.4 支持可变VNF数量

**改进方案:**
```python
# 前端传入VNF数量
resource_request = {
    "vnf_count": 3,  # 新增参数
    "vcpu": 2,
    "memory": 2
}

# 后端处理
vnf_count = resource_request.get("vnf_count", 2)
env = GraphVNEEnv(vnf_count=vnf_count)
```

**需要修改:**
1. 前端: step4/step5 添加VNF数量选择
2. gateway.py: 解析vnf_count参数
3. vne_env.py: 动态设置action_space

**建议优先级: 中**

## 七、总结

当前编排系统存在以下核心问题：

1. **QoS参数无效** - 需要重新设计或移除
2. **环境不感知变化** - 需要动态加载资源
3. **链路未分配** - 核心功能缺失
4. **VNF数量固定** - 灵活性不足

建议按优先级依次修复：
- **最高优先级**: 链路分配 + 动态环境
- **高优先级**: QoS简化 + VNF数量
- **中优先级**: 文档完善 + 清理代码
